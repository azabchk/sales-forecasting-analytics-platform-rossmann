# План работ по проекту ВКР

## 1. Команда и роли

- Разработчик A (ML/Data): DWH, ETL, ML, FastAPI
- Разработчик B (Web): React UI, интеграция API, визуализация

## 2. Этапы и milestones

### Этап 1. Анализ требований и проектирование (Неделя 1)

Результаты:
- зафиксирована цель и границы проекта;
- согласована архитектура;
- подготовлена структура репозитория.

Deliverables:
- `docs/Концепция_ВКР.md`
- `docs/Архитектура.md`

### Этап 2. DWH и SQL-слой (Неделя 2)

Результаты:
- реализована Star Schema;
- добавлены индексы и ограничения;
- созданы KPI views.

Deliverables:
- `sql/01_schema.sql`
- `sql/02_views_kpi.sql`
- `sql/03_indexes.sql`

### Этап 3. ETL и quality checks (Неделя 3)

Результаты:
- реализована загрузка `train.csv`/`store.csv`;
- настроена идемпотентная стратегия;
- реализованы проверки качества.

Deliverables:
- `etl/etl_load.py`
- `etl/data_quality.py`

### Этап 4. ML pipeline (Неделя 4)

Результаты:
- реализован feature engineering;
- обучены Ridge и CatBoost;
- выбран лучший алгоритм;
- сохранены артефакты и model card.

Deliverables:
- `ml/features.py`
- `ml/train.py`
- `ml/evaluate.py`
- `ml/predict.py`
- `ml/model_card.md`

### Этап 5. Backend API (Неделя 5)

Результаты:
- реализованы endpoint-ы `/api/v1`;
- подключены SQL views;
- добавлен сервис прогноза на основе `model.joblib`.

Deliverables:
- `backend/app/*`
- `docs/API_контракт.md`

### Этап 6. Frontend Dashboard (Неделя 6)

Результаты:
- реализованы 3 страницы (Overview, Store Analytics, Forecast);
- подключены API вызовы;
- добавлена визуализация KPI и графиков.

Deliverables:
- `frontend/src/*`

### Этап 7. Интеграция и финальное тестирование (Неделя 7)

Результаты:
- end-to-end запуск на Ubuntu 24.04;
- проверка сценариев и устранение дефектов;
- оформление итоговой документации.

Deliverables:
- `README.md`
- финальный отчет по тестированию (в рамках ВКР)

## 3. Контрольные точки

1. M1: SQL + DWH готов
2. M2: ETL и quality checks пройдены
3. M3: модель обучена, метрики рассчитаны
4. M4: backend API стабилен
5. M5: frontend интегрирован
6. M6: end-to-end demo готов

## 4. Риски и mitigation

1. Риск: низкое качество данных
   - Мера: формализованные DQ-проверки и логирование
2. Риск: переобучение модели
   - Мера: time-based split и сравнение моделей
3. Риск: рассинхронизация API и UI
   - Мера: документированный API-контракт и единые DTO
4. Риск: срыв сроков
   - Мера: короткие итерации и weekly milestones

## 5. Критерии приемки

- Репозиторий запускается локально по инструкции.
- ETL загружает данные в DWH без ручных правок.
- Модель обучается и сохраняет артефакты.
- API возвращает корректные ответы.
- Frontend отображает KPI, таймсерии и прогноз.
- Документация полностью на русском языке.
